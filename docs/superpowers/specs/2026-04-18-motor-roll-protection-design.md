# 电机横滚保护设计

## 1. 背景与目标

当前项目需要在 IMU 横滚角过大时立即切断电机输出，避免车辆侧翻后电机继续输出带来机械冲击和调试风险。

本设计目标：

- 当物理横滚角超过阈值时，强制电机输出为 0
- 增加迟滞恢复逻辑，避免角度在阈值附近抖动时电机频繁开关
- 上电默认处于保护态，防止设备在倾倒状态下上电瞬间误输出
- 在不修改 IMU 驱动层和底层电机驱动层的前提下，以最小侵入方式接入现有工程

## 2. 已确认事实

以下内容来自当前代码库真实上下文：

- 当前工程中变量命名与物理姿态含义相反，`pitch` 表示横滚角，`roll` 表示俯仰角，见 `code/app/imu_app.c`
- IMU 姿态全局量由 `code/app/imu_app.c` 更新，并在 `code/app/imu_app.h` 中对外暴露
- 电机应用层输出入口位于 `code/app/motor_app.c` 的 `motor_set_duty()` 和 `motor_set_speed()`
- 当前仓库内未发现现成单元测试框架，本次验证以编译检查、日志观察和台架行为验证为主

## 3. 设计范围

本次改动仅覆盖电机应用层保护逻辑：

- 修改 `code/app/motor_app.c`
- 视需要修改 `code/app/motor_app.h`，仅用于补充注释或暴露必要声明

本次不修改：

- `code/app/imu_app.c` 的姿态计算方式
- `code/driver/motor/*` 底层驱动协议实现
- 调度器任务结构

## 4. 方案选择

对比过三种落点：

1. 在 `motor_set_duty()` 内直接判断横滚角
2. 在电机应用层统一门控 `motor_set_duty()` 和 `motor_set_speed()`
3. 在调度或上层控制逻辑中手动下发 `motor_set_duty(0)`

采用方案 2。

原因：

- 保护逻辑应作为电机应用层边界能力存在，而不是散落在上层调用方
- 同时覆盖占空比模式和速度模式，避免未来切换控制方式时保护失效
- 保持底层驱动纯粹，不把业务安全策略耦合到驱动层

## 5. 保护策略

### 5.1 角度定义

保护依据使用物理横滚角，对应当前工程变量 `pitch`。

判断量为：

- `roll_abs = abs(pitch)`

### 5.2 阈值

采用迟滞保护：

- 切断阈值：`22.0f`
- 恢复阈值：`18.0f`

要求恢复阈值严格小于切断阈值，用于形成稳定死区。

### 5.3 状态机

定义两个状态：

- 正常态：允许电机输出
- 保护态：强制电机输出为 0

状态切换规则：

- 上电默认进入保护态
- 当处于保护态且 `abs(pitch) <= 18.0f` 时，退出保护态
- 当处于正常态且 `abs(pitch) >= 22.0f` 时，进入保护态
- 其余情况下保持当前状态不变

### 5.4 输出行为

当处于保护态时：

- `motor_set_duty()` 不论传入值为何，最终都下发 0 占空比
- `motor_set_speed()` 不论传入值为何，最终都下发 0 目标速度

当处于正常态时：

- 保持现有输出限幅逻辑不变

## 6. 实现结构

在 `code/app/motor_app.c` 内新增以下内部能力：

- 保护阈值宏
- 一个静态保护状态变量
- 一个静态状态更新函数，用于根据 `pitch` 更新保护状态
- 一个静态日志输出控制逻辑，仅在状态切换时打印一次

建议结构如下：

1. 定义切断阈值与恢复阈值宏
2. 定义静态布尔状态 `motor_roll_protected`
3. 定义静态函数，用于：
   - 读取当前 `pitch`
   - 计算绝对值
   - 执行迟滞状态切换
   - 在状态切换时打印单次日志
   - 返回当前是否处于保护态
4. 在 `motor_set_duty()` 内调用该函数
5. 在 `motor_set_speed()` 内调用该函数

## 7. 日志策略

日志仅在保护状态发生切换时输出一次，不在每周期打印。

日志内容应至少表达：

- 已进入横滚保护
- 已退出横滚保护
- 当前横滚角值

日志目的：

- 便于调车时观察保护是否触发
- 避免高频打印影响串口带宽与实时性

## 8. 边界与风险

### 8.1 命名风险

当前工程中 `pitch` 和 `roll` 的物理含义与常规命名相反，是本次需求中最容易引入错误的点。实现时必须以注释明确说明：保护基于物理横滚角，使用变量 `pitch`。

### 8.2 架构风险

如果后续代码直接绕过 `motor_app` 调用底层 `small_driver_set_duty()` 或 `small_driver_set_speed()`，本次保护将失效。本次不修改该架构，但应将“禁止绕过应用层直接下发电机命令”作为开发约束。

### 8.3 自动恢复风险

本设计采用自动恢复，即姿态回到恢复阈值内后允许重新输出。该策略适合当前调试阶段，但若后续车辆速度更高或风险等级更高，可进一步升级为锁存保护并要求人工复位。

## 9. 验证方案

### 9.1 逻辑验证

重点验证以下场景：

- 上电后若 `abs(pitch) > 18.0f`，应保持保护态
- 上电后若 `abs(pitch) <= 18.0f`，应允许退出保护态
- 正常态下当 `abs(pitch)` 从小于 22 增大到大于等于 22 时，应进入保护态
- 保护态下当 `abs(pitch)` 从大于 18 降到小于等于 18 时，应退出保护态
- 在 `18 < abs(pitch) < 22` 区间内往返变化时，状态不应反复切换
- 正负方向横滚角应对称触发保护

### 9.2 输出验证

- 保护态下调用 `motor_set_duty(正值)`，底层实际下发应为 0
- 保护态下调用 `motor_set_duty(负值)`，底层实际下发应为 0
- 保护态下调用 `motor_set_speed(非零)`，底层实际下发应为 0
- 正常态下应保持原有限幅行为

### 9.3 日志验证

- 首次进入保护态时仅打印一次进入日志
- 在保护态持续期间不重复打印进入日志
- 退出保护态时仅打印一次退出日志
- 在 18 到 22 度区间内晃动时不应反复刷日志

### 9.4 台架验证

- 人工倾斜车体超过切断阈值，观察电机输出被压为 0
- 角度回到恢复阈值内，观察电机输出恢复
- 多次跨越阈值，观察日志与输出行为一致

## 10. 后续建议

以下建议不纳入本次实现范围，但值得后续考虑：

- 将保护切换为锁存模式，触发后需人工复位
- 给保护逻辑增加总开关，便于实验阶段快速启停
- 在上位机或调试串口中增加保护状态显示
- 系统内统一姿态变量命名，减少 `pitch`/`roll` 语义反转带来的维护风险
